# 博客重建设计说明

- 日期：2026-04-14
- 仓库：`yinshaojun001.github.io`
- 目标：把当前仅包含静态发布产物的博客仓库，重建为一个真正可维护的 Hugo 博客工程，并恢复自动发布能力。

## 1. 当前现状

### 1.1 已确认事实

当前仓库并不是可直接维护的 Hugo 源码工程，而是 Hugo 生成后的静态站点结果。依据如下：

- 仓库根目录以 `index.html`、`posts/.../index.html`、`assets/css/...` 等静态文件为主
- 未发现 Hugo 源码工程常见目录与配置：
  - `content/`
  - `layouts/`
  - `themes/`
  - `hugo.toml` / `config.toml`
- 页面生成信息显示该站点此前由：
  - Hugo `0.147.8`
  - PaperMod
  生成
- 当前 `master` 分支保存的是已发布内容，`main` 分支当前仅有一个 `README.md`

### 1.2 推断

用户此前大概率在另一台电脑或另一份本地目录中维护 Hugo 源码，再把构建结果发布到当前仓库。由于换电脑，原 Hugo 源码工程已缺失，目前仓库只剩部署结果。

## 2. 重建目标

### 2.1 核心目标

重建一个可长期维护、可持续打磨、支持自动发布的新博客工程。

### 2.2 用户已确认的方向

- 博客定位：**个人介绍 + 技术输出**
- 首页取向：**偏技术博客，最新文章优先，个人介绍作为辅助**
- 新风格方向：**个人品牌型博客**
- 旧文处理：**尽量全部迁移**
- 发布方式：**GitHub 自动发布**
- 后续维护方式：**作为作品长期打磨，愿意修改模板、样式、结构**

## 3. 设计原则

1. **先恢复可维护性，再逐步美化。**
2. **优先保留内容资产。** 旧文章应尽量迁回可编辑的 Markdown。
3. **内容优先于炫技。** 首页需要有设计感，但不能牺牲技术博客的阅读体验。
4. **结构要能演进。** 第一版不能过重，但必须为后续自定义留出空间。
5. **发布流程自动化。** 用户以后应只需维护源代码与 Markdown 内容。

## 4. 方案比较与最终选择

### 4.1 评估过的方案

#### 方案 A：标准 Hugo + 成熟主题二次定制

优点：

- 启动快
- 维护成本低
- Hugo 原生能力可快速恢复

缺点：

- 主题痕迹较重
- 个性化上限较低

#### 方案 B：Hugo 做内容引擎 + 自定义首页与关键模板

优点：

- 在可维护性与个性化之间平衡最好
- 既适合内容生产，也适合长期打磨为个人作品
- 可以先做骨架，再持续增强视觉与结构

缺点：

- 首次重建工作量高于纯套主题
- 需要自行承担部分模板与样式设计

#### 方案 C：完全自定义博客前端，仅借 Hugo 生成内容

优点：

- 自由度最高
- 作品感最强

缺点：

- 初始成本最高
- 不利于尽快恢复写作和发布能力

### 4.2 最终选择

采用 **方案 B**：

> **Hugo 做内容引擎，自定义首页与关键模板。**

这是当前最符合用户目标的折中方案。

## 5. 站点结构设计

### 5.1 第一版导航

第一版站点导航定为：

- `Home`
- `Posts`
- `About`
- `Tags`

另：

- `Notes` 作为后续扩展能力保留，但不要求在首版上线

### 5.2 首页结构

首页信息层级：

1. 顶部导航与简洁品牌区
2. 一句个人定位 / 当前关注方向
3. 最新文章与置顶文章列表（首页主角）
4. 简短个人介绍卡片（辅助）
5. 标签 / 技术栈 / 后续扩展区

### 5.3 页面角色

#### Home

- 承担首页展示与品牌入口
- 强调最新内容与重点内容
- 不做纯个人主页，不压过文章内容

#### Posts

- 所有正式文章的归档与列表页
- 可按时间顺序展示
- 后续可加入系列、筛选、专题入口

#### About

- 单独承载完整个人介绍
- 首页只保留简介摘要

#### Tags

- 作为技术博客的主要主题导航方式
- 方便未来内容积累后按主题回看

## 6. 内容模型设计

### 6.1 主内容类型

第一版内容类型保持精简：

- `posts`：正式文章
- `about`：个人介绍页

### 6.2 正式文章组织方式

所有正式文章统一放在 `posts` 下，通过以下方式组织：

- `tags`
- `series`（后续增加）

不在第一版一开始就拆太多栏目，以避免早期结构过重。

### 6.3 旧文迁移原则

迁移时尽量恢复：

- 标题
- 日期
- 正文
- 标签 / 分类（若可提取）
- 原有 slug / URL 兼容性

若个别文章存在元信息缺失，则优先：

1. 保住正文
2. 再补 front matter

## 7. 技术架构设计

### 7.1 技术选型

- 内容引擎：**Hugo**
- 模板策略：**自定义关键页面模板**
- 部署：**GitHub Pages**
- 自动化：**GitHub Actions**

### 7.2 分支策略

目标分支结构：

- `main`：Hugo 源码分支
- `gh-pages`：静态发布结果分支（或 GitHub Pages 官方工作流产物）

说明：

- 用户以后主要维护 `main`
- 发布结果不再作为人工维护对象

### 7.3 首版自定义范围

首版重点自定义：

- 首页模板
- 文章列表页
- 单篇文章页
- About 页
- 公共头部 / 页脚 / 基础 SEO 结构

首版暂缓的能力：

- 搜索
- 评论
- 多语言
- 重动画
- 复杂交互模块

## 8. 迁移与实施策略

采用“两阶段重建”：

### 阶段一：先搭新工程骨架

目标：

- 新 Hugo 工程可运行
- 本地可预览
- 自定义首页与基础页面可访问
- 自动发布链路打通
- 有至少一篇示例文章跑通

### 阶段二：旧文批量恢复

目标：

- 从现有 HTML 中尽量提取文章
- 恢复为 Hugo 可维护的 Markdown 源文件
- 校验标题、日期、slug、正文是否正确
- 修复少数恢复不完美的文章

## 9. 风险与应对

### 风险 1：旧文章无法完美恢复为 Markdown

应对：

- 先做自动提取
- 再针对少数格式异常文章人工修复

### 风险 2：现有链接结构变化导致历史链接失效

应对：

- 尽量复用旧 slug
- 必要时增加 alias / redirect 兼容旧链接

### 风险 3：首版范围失控

应对：

- 只完成最关键页面
- 把搜索、评论、复杂视觉增强放到后续迭代

## 10. 首版验收标准

首版重建完成应满足：

1. 仓库中存在完整的 Hugo 源码工程结构
2. 本地可通过 Hugo 正常启动预览
3. 首页、Posts、About、Tags 页面可访问
4. GitHub Actions 可自动构建并发布
5. 至少一篇新文章可通过 Markdown 正常发布
6. 旧站文章可开始批量迁入新工程

## 11. 本次设计结论

本项目不再沿用当前“直接维护静态 HTML”的方式，而是重建为：

> **一个以 Hugo 为内容引擎、以自定义关键模板为展示层、以 GitHub Actions 为自动发布通道的新博客工程。**

第一版的重点不是一次性做满，而是：

1. 恢复可维护性
2. 恢复写作能力
3. 恢复自动发布
4. 为后续长期打磨保留足够空间
